Module Description
Software Development Basics
covers the
fundamental skills needed by any software engineer
. It goes much beyond simply learning basic programming but also covers the entire software development environment and workflow.
You'll learn to turn software requirements into concrete tasks, work iteratively, understand and debug errors, and use version control and documentation in the process.
📚 How this module works
It's recommended to take this module
early in your studies
as it covers topics you will likely require knowing in most other software engineering modules.
It's a common misconception to think you could learn programming and related skills by simply understanding concepts from reading a tutorial or attending a class.
🛠️
But the most effective way to learn the contents of this module is only through practice and repetition!
The
📚 learning resources
provide you with a good and important starting point. They give you a complete overview of all the relevant basics you should know before the assessment.
This module also offers introductory 👩‍🏫
learning units
to help you know what to practice.
However, you will only learn the contents of this module by applying them in projects and practicing them.
📌 Prerequisites to this module
This module is a continuation of OS_01 (Introduction to Software Engineering). Therefore, coming into this module, you should have base-level programming skills in at least one programming language (e.g., in Python) and know how to set up new code projects in different folders on your computer.
❓ Frequently asked questions
\"Can I just use any code project to pass this module?\"
Please take a look at the \"alternative assessment\" section. Generally, it's possible to pass this module on to any intermediate code project. However, it must be clear through the git commit history that you have personally worked on many smaller changes.
\"How do I know my git commit history is appropriate to pass?\"
If a significant amount of code is added through a single large commit, there is a very high likelihood of failing the assessment. Ensure to use commits throughout the development process, showing that you have personally changed the code over time and showing your general learning progression. Neither the commit messages nor the committed code need to be perfect. The important thing is to see progression and code changes over time.
\"Can I use generative AI like
ChatGPT
for my project?\"
Not for this module, no. This module covers the absolute basics of software engineering. Those are fundamental skills needed no matter if you work with or without generative AI in software projects later on.
Level Expectations
This is a pass/fail module. To pass this module, projects must meet
all qualification objectives
listed below.
Understand and use the basic features of programming languages
(which includes functions, loops, conditions, arrays, etc.)
Use the command-line interface (CLI)
Apply basic version control
(usually git)
Use integrated development environments (IDE)
Understand iterative development
Debug and troubleshoot computer programs
Details and contents of each of these objectives are outlined in the
📚 learning resources
.
❗️ Only
your competency
is evaluated. Always be transparent if any parts of your code didn't come from you personally.
Standard Assessment Instructions
The standard assessment is an
in-person exam
. It consists of a small code project that can be completed within 1–2 hours. However, students may take up to 4 hours if they need more time. Their goal is to build a small game or app. The exact instructions will be revealed at the time of the exam.
The goal of the exam is for students to demonstrate their competency in solving a task using code and git as version control. Keeping that goal in mind, students ...
... must use
Git
to track their progress in granular commits and submit the final result to a remote repository.
... may choose any programming language, library, or game engine.
... may use the language or library documentation as a reference.
...
are not
permitted to use generative AI during the exam.
The assessor will be present during the exam and available to answer questions and offer some pointers if students get stuck.
💻 Students are expected to
bring their own laptop
with the necessary tools pre-installed to complete the exam (IDE, git, chosen programming language interpreter/compiler). If that's not possible, please inform the assessor at least 48 hours before the exam.
Alternative Assessment Instructions
There are two types of alternative assessments available for this module:
Hand in an intermediate code project with a granular git commit history.
A combined assessment confirmed by another module's assessor.
👉 If you wish to be assessed using an
alternative assessment
as opposed to the standard assessment,
you must reach out to the module coordinator before the module registration deadline
.
🤖 Only code that has not been generated by generative AI can be used for the SE_01 assessment.
1) Intermediate Project Hand-in
Since SE_01 covers the basic skills needed for most software engineering tasks, most intermediate software engineering projects include this module's qualification objectives. Those could be projects from your job or another CODE project. Talk to the module coordinator to find out if your project is suitable. Keep in mind that a clean git commit history of your personal work is a hard requirement for this type of assessment.

Early Assessment Instructions
An Early Assessment is available for this module. An early assessment is available
any time during the project phase
and
before the module registration deadline
. Find the exact dates on the
academic calendar
.
To sign up for an early assessment, follow these steps:
Prepare all hand-in material as described in the
standard assessment
(above). Check the
level expectations
(below) to ensure you've met all the requirements.
During the project phase, contact the module coordinator (e.g., on Slack) to inform them you'd like to take an early assessment.
Create an early assessment proposal in the learning platform. Use the \"standard\" assessment type unless you explicitly discussed something else with the module coordinator.
Together with the module coordinator, you now agree on a hand-in deadline and time and date of the oral exam. Afterward, the module coordinator enters those into the learning platform.
Now, you're able to submit your hand-in in the learning platform and are all set.
